DeclarativeBase Vs declarative_base
    declarative_base() -> old style(functions that returns a base class)
    DeclarativeBase -> new style (explicts base class we inherit from)
    Both are from SQLAlchemy, but, DeclarativeBase is the mordern, recommended approach(SQLAlchemy 2.0.1)
    Side-by-Side Comparision 
        Old way: declarative_base()
            What's happening?
                We call a function 
                It dynamically creates a base class 
                We inherits from that 
            Feels a bit "magical"
        New way: DeclarativeBase
            What's happening?
                We explictly define a base class 
                Much clearer and more type-safe
    Why DeclarativeBase Exists?
        The new approach improves 
            Type hinting 
                Better support for:
                    IDE autocomplete 
                    Static analysis(mypy, pyright)
            Explictness 
                Instead of : Base = declarative_base()
                we now have: class Base(DeclarativeBase): pass 
                more pythonic, less "magic"
            Extensibility 
                We can customize our base class 
                And all models will inherit this behaviour automatically 

Difference between using Column and mapped_column()
    Column -> Old style (classic SQLAlchemy)
    mapped_column -> new style (SQLAlchemy 2.0, type-safe)
    Both are from SQLAlchemy, but, mapped_column is designed to work with python hints(Mapped[])
    Side-By-Side Comparision
        Old way: Column 
            Problems:
                No real type safety 
                IDEs/mypy struggle 
                Types are duplicated(Python Vs SQLAlchemy)
        New way: mapped_column()
            Much cleaner:
                Types comes from python(int, str)
                SQLAlchemy infers the database logic 
    Core difference
        Column(old mental model)
            id = Column(Integer)
            This is a database column 
        mapped_column(new mental model)
            id: Mapped[int] = mapped_column()
            This is a python mapped to database column 
    Why mapped_column is better?
        Type Safety 
            name: Mapped[str]
            works perfectly with:
                mypy 
                pyright 
                IDE autocomplete
        Less Duplication 
            No need to repeat types in two systems 
        Cleaner models 
            We can even ommit mapped_column()
            SQLAlchemy infers everthing
        Designed for SQLAlchemy 2.0 
            This is the official mordern pattern 
    mapped_column() is the mordern, type-safe replacement for Column, designed to work seamlessly with python hints in SQLAlchemy 2.0

Redis 
    What is Redis? - Overview of Redis 
        Redis is an open source, in-memory data structure store that can be used as a database, cache, message broker, and streaming engine. 
        It belongs to the class of NoSQL Database know as key-value stores 
        Redis is widely adopted for its sub-milliseconds response time. rich data structures and versatility across use case from caching to real-time analytics 
    How does Redis works?
        Redis store all data in memory as key/value pairs, where each key is a unique identifier mapped to a typed value
        Each data type has its own set of commands for reading and writing data 
        For example: we can store a name in Redis string and associate it with the key "myname" using the SET command, then retrieve using GET 
            SET myname "Kiran Khayamali"
                OK 
            GET myname 
                "Kiran Khayamali"
    What are Redis Data types?
        Redis goes for beyond simple key/value string storage 
        It supports a rich set of data types, each optimized for specific access pattern 
            String 
                The simplest type, stores text, numeric, or binary data up to 512 MB 
                Example: Counters, cached values, session tokens 
            Lists 
                Ordered collection of strings(Linked lists)
                Example: Activity feeds, message queues 
            Sets 
                Unordered collections of unique strings 
                Example: tags, unique visitors, set operations 
            Hashes 
                Mapps of field-value pairs, similar to objects 
                Example: User profiles, product details 
            Sorted Sets 
                Sets ordered by a numeric scores 
                Example: Leaderboards, priority queues, rate limiters 
            Streams 
                Append-only log data structures 
                Example: Event sourcing, activity logs 
            JSON 
                Native JSON document storage and querying 
                Example: Complex nested data, API Response caching 
            Vector Sets 
                Elements associated with vector embedding for similarity search 
                Example: Semantic search, recommendations, RAG 
        Redis also supports specialized types like: Bitmap, Bitfield, Hyperloglog, and Geospatial Indexes for advanced use cases 
    What is Redis used for?
        Redis is used across industries wherever fast data access matters 
        Common use cases includes 
            Caching: Store frequently accessed data in memory to reduce database load and speed up response time 
            Session management: Maintain user session data across distributed web applications with built-ing expiration 
            Leaderboards and rankings: Use Sorted Sets to build real-time leaderboards with automatic ordering by score 
            Real-time analytics: Tracks metrics, counts and time-series data with sub-milliseconds writes 
            Pub/Sub messaging: Broadcast messages to muliple subscribers for real-time notifications and chat systems 
            Rate limiting: Control API usage and prevent abuse using couters with expirations 
            Queues and background jobs: Use Lists and streams as lightweight, high-performance task queues
            GenAI and vector search: Store vector embedding and run similarity searches for semantic search, context retrival, agent memory, retrieval-augmented generation(RAG), and recommendation systems 
    Advantages of Redis? 
        Redis combines in-memory speed with rich data structures, persistence, and horizontal scalability 
        Key advantages include:
            Sub-milliseconds latency: All data lives in RAM, so reads and writes completes in microseconds rather than milliseconds
            Rich data structures: Go beyond simple key/value strings with Lists, Sets, Hashes, Sorted Sets, Streams, JSON, and more.
            Built-in persistence: RDB snapshots and AOF logs keeps our data durable without sacrificing performance 
            Horizontal scalability: Redis cluster distributes data across multiple nodes with automatic sharding and failover 
            Broad language support: Official clients libaries exists for Python, Node.js, Java, Go, .NET, and many other languages
            Vector search and GenAI: Native support for vector embeddings, similarity search and hybrid queries makes Redis a strong fit for GenAI applications 
            Versatile use cases: Use Redis as a cache, primary database, message broker, vector database, streaming engine, or all of the above in a single deployment 
            Active ecosystem: A large open-source community, comprehensive docs, and Redis University courses make it easy to learn and adopt
    How fast is Redis?
        Redis is one of the fastest databases available 
        Because it stores and serves all data from RAM rather than disk, typical operations complete in under one milliseconds - often in the range of single-digit microseconds 
        A single Redis instance can handle over 100,000 operations per second on modest hardware and benchmarks on optimized configurations regularly exceed 1 millions operations per seconds 
        This speed makes Redis an excellent choice for applications that requires real-time data access, such as gaming leaderboards, financial trading platforms, and ad-tech bidding systems 
    How does Redis persist data?
        Redis persists data to disk so it survives restarts, even though all reads and writes happen in memory 
        It provides multiple persistence options to ensure durability 
            RDB(Redis Database): Takes point-in-time snapshots of our data at configured intervals 
            AOF(Append Only File): Logs every write operation, providing stronger durability guarentees 
            RDB + AOF: Combines both methods for a balance of performance and data safety 
        This means our data is durable even though all reads are served from an in-memory copy, giving us both speed and reliability
    Using Redis with FastAPI 
        FastAPI is a python web framework based on the starlette microframework 
        With deep support for asyncio, FastAPI is indeed very fast 
        FastAPI als distinguishes itself with features like automatic OpeanAPI(OAS) documentation for our API, easy-to-use data validation tools, and more 
        Of course, the best way to make our FastAPI service even faster is to use Redis 
        Unlike most databases, Redis excels at low-latency access because it's an in-memory database
    Why use Redis for time series data? 
        Redis support time series data. Time series is a great way to model any data that we want to query over time.(ie ever-changing of stocks)

    What is token-based authentication?
        Token-based authentication is a pattern where the server issues a token after a user login 
        The client sends that token as a bearer token with every subsequent request 
        The server validates the token, identifies the user, and decides whether to allow the request 
        The server needs a fast way to validate, rotate, and revoke token, That is where Redis fits, It stores the live token state so every request gets a sub-milliseconds lookup, and TTL handles cleanup automatically
    Why use Redis for authentication token storage?
        Redis is a strong fit when we need fast token lookup and instant revocation 
        A database can store token state, but Redis gives us low-latency reads, built-in TTL-based expiration, and single command invalidation with DEL 
    Authentication token storage with Redis (Tutorial follows bun and docker)
        How does the app store tokens in Redis?
            The application uses two key prefixes:
                1. Key: token{uuid}     Value: The username that owns the token     TTL: TOKEN_TTL(default 3600sec)     Redis Command: SET ....... NX 
                2. Key: users{username}     Value: JSON string with username, password hash, and created date   TTL: None   Redis Command: SET ...... EX 
            Token validation is a single GET: 
                Lookup tokens:{token} in Redis 
                If the key exists, the value is the username - the token is valid 
                If the key is missing, the token has either expired(TTL removed it) or been deleted(logout removed it). Return 401 
            This is the core Redis pattern. Every token check is one GET with O(1) complexity, and Redis removes expired tokens automatically 
        How does registration work?
            POST /api/auth/register creates a new user account. The application hashes the password with Bun.password.hash() and stores the user data in Redis with SET .... NX 
            The NX flag tells Redis to set the key only if it doesn't already exist. This prevents duplicate accounts atomically - no seperate "check then insert" step that could race 
            Under the hood, Redis stores: 
                SET users: kiran '{"username": "kiran", "passwordHash": "..........", "createdDate": "............"}' NX 
        How does login work ?
            POST /api/auth/login accepts a username and password. The application reads the user record from Redis with GET, verifies the password has with Bun.password.verify(), and generates a UUID token. It stores the token in Redis with SET ..... EX, so the token expires automatically after TOKEN_TTL seconds
            Under the hood, Redis stores: 
                SET tokens: <token> kiran EX 3600 
            The client includes this token in all subsequent requests as Authorization.Bearer......
        How does request validation work?
            Every protected route runs throught the requireAuth middleware. The middleware: 
                Extracts the bearer token from the Authorization header 
                Calls GET tokens:{token} in Redis 
                If the key exists, the value is the authenticated username. The request continues 
                If the key is missing(expired or deleted), the middleware returns 401 
        How does token refresh work?
            POST /api/auth/refresh rotates the token. The middleware validates the current token, then the handler deletes the old key and creates a new one with a fresh TTL 
            The steps: 
                GET tokens:{old_token}  -> retrieve the username 
                DEL tokens:{old_token}  -> remove the old token 
                SET tokens:{new_token} username EX 3600     -> store the new token with a fresh TTL 
        How does logout work?
            POST /api/auth/logout dletes the token from Redis with a single DEL. The next request using that token get nil from GET and the middleware returns 401 
            No revocation markers, no cleanup jobs. DEL removes the key immediately and the token is invalid from that moment forward
