File Structure 
    Core Principle(Before Structure)
        Think in terms of seperation of concerns 
            API layer -> handles HTTP(router)
            Service layer -> business logic 
            Data layer -> database interaction 
            Schemas -> validation (pydantic)
            Models -> database structure(ORM)
        If we mix these, our code becomes painful very quickly 
    Recommended Project Structure(From Chatgpt)
        Here's solid, production ready layout 
            schemas/
                user.py 
                item.py 
            
            services/
                user_service.py 
                item_service.py 
            
            repositories/
                user_repository.py 
                item_repository.py 

            db/
                base.py 
                session.py 
                migrations/
            
            test/ 
                test_users.py 
                test_items.py

        Breakdown 
            main.py -> Entry Point 
                Keep it thin, No logic here 
            api/ -> Routes Layer 
                Only HTTP concerns 
                    Request 
                    Response 
                    Status Code 
                No database queries here 
            schemas/ -> Pydantic Models(Validation)
                Used for 
                    Request Validation 
                    Response Validation 
            models/ -> Database Models(ORM)
                These represents actual database tables 
            services/ -> Business logic 
                This is where 
                    Rules lives 
                    Logic grows 
                    Complexity belong 
                If our project get messy, it usually because this layer is missing 
            repositories/ -> Database Access 
                Only responsibility 
                    CRUD operations 
                    Queries 
                No business logic 
            core/ -> Shared infrastructure
                config.py -> enviroment Variables
                security.py -> auth helpers (JWT, hashing)
                dependencies.py -> reusable FastAPI dependencies
            db/ -> Database setup 
                session.py -> Database session 
                base.py -> Base Class 
                migrations -> Alembic 
            tests/ -> Tests 
                for testing the models and features 
            Optional(But useful as we grow)
                We can later add 
                    utils/
                    exceptions/ 
                    middlewares/
        Common mistakes
            Putting everthing in routes 
                This becomes unmaintainable fast 
            Skipping service layer 
                Leads to duplicated logic 
                Hard to test 
                Tight coupling 
            Mixing Schemas and Models 
                Causing confusion 
                Breaks seperation of concerns 
        Mental Model(Remember This)
            Think:
                Request -> API -> Service -> Repository -> Database 
                                    ^
                                    |
                                    Schemas
        Advice 
            Start Simple, but structure early 
            Enforce boundaries(routes != logic != database)
            Name things clearly( user_service, not utils)
            Refractor agressively as complexity grows 
                
DeclarativeBase Vs declarative_base
    declarative_base() -> old style(functions that returns a base class)
    DeclarativeBase -> new style (explicts base class we inherit from)
    Both are from SQLAlchemy, but, DeclarativeBase is the mordern, recommended approach(SQLAlchemy 2.0.1)
    Side-by-Side Comparision 
        Old way: declarative_base()
            What's happening?
                We call a function 
                It dynamically creates a base class 
                We inherits from that 
            Feels a bit "magical"
        New way: DeclarativeBase
            What's happening?
                We explictly define a base class 
                Much clearer and more type-safe
    Why DeclarativeBase Exists?
        The new approach improves 
            Type hinting 
                Better support for:
                    IDE autocomplete 
                    Static analysis(mypy, pyright)
            Explictness 
                Instead of : Base = declarative_base()
                we now have: class Base(DeclarativeBase): pass 
                more pythonic, less "magic"
            Extensibility 
                We can customize our base class 
                And all models will inherit this behaviour automatically 

Difference between using Column and mapped_column()
    Column -> Old style (classic SQLAlchemy)
    mapped_column -> new style (SQLAlchemy 2.0, type-safe)
    Both are from SQLAlchemy, but, mapped_column is designed to work with python hints(Mapped[])
    Side-By-Side Comparision
        Old way: Column 
            Problems:
                No real type safety 
                IDEs/mypy struggle 
                Types are duplicated(Python Vs SQLAlchemy)
        New way: mapped_column()
            Much cleaner:
                Types comes from python(int, str)
                SQLAlchemy infers the database logic 
    Core difference
        Column(old mental model)
            id = Column(Integer)
            This is a database column 
        mapped_column(new mental model)
            id: Mapped[int] = mapped_column()
            This is a python mapped to database column 
    Why mapped_column is better?
        Type Safety 
            name: Mapped[str]
            works perfectly with:
                mypy 
                pyright 
                IDE autocomplete
        Less Duplication 
            No need to repeat types in two systems 
        Cleaner models 
            We can even ommit mapped_column()
            SQLAlchemy infers everthing
        Designed for SQLAlchemy 2.0 
            This is the official mordern pattern 
    mapped_column() is the mordern, type-safe replacement for Column, designed to work seamlessly with python hints in SQLAlchemy 2.0

Tracing the boilerplate of benavlabs
    Big Picture 
        The boilerplate code follows a layered architecture
            Client -> API -> CRUD(business logic) -> Models(database) -> Database
                               ^
                               |
                            Schemas(validations)
            This seperation is intentional:
                API = handles HTTP 
                CRUD = Logic 
                Models = database structure
                Schemas = validation and serialization 
            This is called seperation of concerns 
    Root Structure(What matters Vs What doesn't)
        FastAPI-boilerplate/
            src/    <- our main code 
            tests/  <- tests
            docs/   <- documentation 
            Docker stuff    <- Deployment 
            
            We can initially ignore the docker and spend 90% of our time in src/ 
    Inside src/ (core of the system)
        src/
            app/
            migrations/
            scripts/

        src/    <- The heart of everthing
            app/
                main.py
                api/
                core/
                crud/
                models/
                schemas/
                middlewares/
            
        Let's break this:
            main.py -> Entry Point 
                This is where the app starts 
                Think: app = FastAPI()
                But in this boilerplate:
                    uses app factory pattern 
                    loads configurations 
                    register routes 
                    adds middlewares
                This is the bootloader of our backend 
            api/ -> where HTTP lives
                api/v1/
                    users.py
                    posts.py 
                    login.py 
                This is what or frontend talks to
                Example: @router.post("/users")
                         async def create_users():
                But: 
                    No heavy logic here 
                    Only: 
                        request validation 
                        calling CRUD 
                        returning response 
                Rule: API layer should be thin  
            schema/ -> Data Contracts 
                These are pydantic models 
                Example: UserCreate, UserRead, UserDelete, UserUpdate 
                Why schemas exists?
                    validate input 
                    control output 
                    prevent security issues
                They sit between API and CRUD 
            models/ -> Data Structure 
                These are SQLAlchemy models 
                Tis maps directly to database tables 
            crud/ -> Business logic layer 
                This is where:
                    database queries happen 
                    logic lives 
                    reusable operation exists 
                This project uses FastCRUD
                    Generic CRUD generator 
                    saves us from rewriting queries 
                This is the service layer 
            core/ -> the Brain 
                core/ 
                    config.py 
                    security.py 
                    db/
                    exceptions/
                This is where system-wide logic lives 
    Important Part:
        config.py 
            Reads .env 
            Stores settings 
            Single source of truth 
        security.py 
            Handles 
                JWT Token 
                Password Hashing 
                Auth logic 
            Critical for authentication
        db/ 
            Database connections 
            session management
            used via dependency injection 
        exceptions/
            Custom errors 
            Centralized error handling 
    DataFlow 
        Let's trace a real request 
            Example: Create User 
                Client request (POST /users)
                API layer (create_users()) ---> validate using schemas 
                Convert to internal schemas (password hashing)
                CRUD layer 
                Model -> Database (sqlalchemy saves to database)
                Response 
            Final Flow 
                Request -> Schema -> API -> CRUD -> Model -> Database 
                Response <- Schema <- API <- CRUD <- Model <- Database
    Advanced Features 
        This isn't just CRD - it includes:
            Authentication
                JWT access + refresh tokens 
                Cookie-based security 
            Async everywhere 
                FastAPI + async SQLAlchemy
            Background Jobs 
                uses Redis + ARQ 
            Rate Limiting 
                Per user tier 
            Caching 
                Redis caching layer 
            Deployment-ready 
                Docker + NGNIX + Gunicorn 
    Design Principles (Why it's built this way)
        Seperation of Concerns 
            API != Database != Logic 
        Modularity 
            We can 
                Remove auth 
                Replace database 
                Add features 
            Scalablity 
                Async 
                Background workers 
                Caching
            Maintainability 
                Easy to debug 
                Easy to extend 
            
    Common Mistakes 
        Putting logic in API 
        Mixing schema + model 
        Direct database queries in routes 
        Ignoring internal schemas 
    Final mental model 
        If we remember just one thing: 
            API talks -> Schemas validate -> CRUD Thinks -> Models store
